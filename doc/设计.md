
使用 rust 实现一个git 托管服务器。

参考 cgit 和 gitea 实现一个git 托管服务器。

有登陆界面。支持群组和子组的层级概念，可分配用户在哪些群组，仓库。群组内的用户可以看到该群组下的所有仓库，但是默认权限不能穿透看到子组，子组需要重新授权。在群组中添加成员时，有一个选项，可以设置是否允许查看子组。可以对单个仓库进行成员管理。

管理员可以访问所有仓库。

其实就是直接管理 repos 目录和子目录。创建群组就是创建一个目录，目录名就是群组名。

比如
* Teams
  * Team1
    * Project1
      * repo1
      * repo2
  * Team2

在 Team1 目录下可以创建仓库目录。

但是通过 sqlite 数据库来管理用户、群组、仓库、权限等信息。主要是用来管理和追踪仓库和群组以及权限等信息。支持直接将仓库复制到或新建文件夹即代表创建组，但是这时数据库还没记录该数据库，所以数默认权限除了管理员，其它人员不可见。start 等信息也空。当然也支持web上创建组和仓库。

仓库组之间可以任意移动，在 web 中移动会更新数据库，但是直接在文件管理中移动则会导致数据库不一致，导致对应仓库或组的数据重置。

为了便于追踪和支持线下移动和整理，我觉得每个组文件夹可以添加 .meta 文件夹，用于管理该组的信息以及组下的仓库信息，不用统计到子组，因为子组有自己的 .mate 然后每个仓库或组也同样分配一个文件夹用来存放数据库或数据文件。这样即使线下移动仓库或组，只要将其对应的 .mate 里面的对应文件夹也一起移动则可以让系统识别之前的信息。比如组的成员信息，仓库的issue信息，仓库成员信息，评论，start 以及 release 版本信息和文件等。这样组织数据库会让仓库具有很高灵活性。而当前data/gitclub.db 数据库主要管理的是用户的信息。

只有组内才有 .meta 仓库内没有. 这是为了保持仓库的原始性，不应该修改仓库任何内容，因为仓库是最终要的内容，不能随意更改。希望的是 组内的 .meta/ 文件夹下有对应仓库名称 比如 a.git b.git  那么有  .meta/a.git/    .meta/b.git/ 组本身信息则放在 .meta/.meta/ 下面。不用统计到子组，因为子组有自己的 .mate

需要添加用户。进入需要登陆，登陆后就可以进入。管理员在配置里面设置。默认 admin 密码 admin123456 支持配置。然后可以注册，支持设置名称，邮箱，登陆密码。然后注册成功会生成自动头像，个人主页可以更改个人信息，签名等。然后会在repos 中分配一个文件夹给用户。管理员本身也有一个文件夹，每个人一个文件夹。每个用户只能看到自己文件夹内仓库，除非仓库或组设置了权限。db 里面记录用户以及用户的权限信息。用户仓库管理除了列出的自己的以自己名字开头，还有组级别的开头。就是自己被授权了那些组或仓库。管理员则可以看到列出的所有用户和组。
